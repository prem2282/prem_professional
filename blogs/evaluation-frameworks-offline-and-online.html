Evaluation Frameworks (Offline and Online)

Offline eval tells you if a release is safe to ship. Online eval tells you if it was safe to ship to today's users. You need both—and they must disagree sometimes, or one of them is lying.

We once green-lit a prompt change that gained 4 points on our golden-set accuracy. Thumbs-down rate in prod rose 12% within 48 hours. The golden set was stale, English-only, and missing the "angry paraphrase" cluster our users actually typed.

Evaluation is how you turn "it feels better" into an engineering decision. Without it, every prompt tweak is a coin flip; with it, you still can be wrong—but you will know where you are wrong before customers do.

Offline vs online: different jobs

Offline: Repeatable golden prompts, human or LLM-judge labels, CI gates on regressions. Optimized for comparability between commits.

Online signals

  • Thumbs up/down and explicit "wrong answer" reports
  • Escalation to human agent within 5 minutes of bot reply
  • Abstain rate drift vs 7-day baseline
  • Canary cohort A/B on task completion (not just click-through)
  • Tool error rate and retry storms on agent flows

Playbook: minimum viable eval program

Golden set hygiene. 150–400 tasks stratified by intent, language, and failure mode. Version it like code.
Primary metric + guardrails. One north-star (e.g. task success); secondary caps on cost, latency, abstain rate.
CI on every prompt/schema change. Block merge when primary drops >2% absolute vs main.
Online shadow & canary. 5% traffic, compare thumbs and escalation rate before full rollout.
Human review queue. Sample low-confidence and high-impact sessions weekly—feed back into golden set.

Metrics that actually correlate with pain

  • Task success (human rubric)—expensive, trustworthy.
  • Faithfulness / groundedness—for RAG; judge with cite-check automation.
  • Tool call success rate—for agents; parse JSON validity + backend HTTP outcome.
  • Escalation & override rate—proxy for silent failure in enterprise UX.

LLM-as-judge: useful, dangerous as default

Good for: Offline batch scoring, regression triage, drafting rubrics, comparing two answers when human lag is too high.
Bad for: Sole production gate on regulated decisions, scoring without golden references, judges with no inter-rater calibration vs humans.
Practice: Freeze judge prompt version; measure disagreement rate with 30 human-labeled samples quarterly.

# ci_eval.yaml (conceptual)
gates:
  primary: task_success
  thresholds:
    task_success: { min_delta_vs_main: -0.02 }
    faithfulness: { min_delta_vs_main: -0.03 }
    p95_latency_ms: { max_delta_vs_main: 150 }
on_fail: block_merge
on_pass: allow_canary_5pct

Human review loop (closed circuit)

  1. Sample 50 sessions/week: low confidence, thumbs-down, high-impact intents.
  2. Label failure mode (retrieval, grounding, tool, UX).
  3. Add mispredictions to golden set within 48 hours—version bump.
  4. Re-run offline eval before next prompt deploy.

Your implementation challenge

Pick one production regression from last month. Reproduce it in a golden row. Wire CI so that row fails on main today and passes only with your fix branch. That single test is worth more than fifty generic BLEU scores.
