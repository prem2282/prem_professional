Function Calling and Tool Use Patterns

Tool-enabled agents are where demos become systems. Once your model can call tools, you are coordinating side effects, retries, and safety boundaries. This guide focuses on practical execution patterns I use for reliable function-calling workflows in production.

What changes when a model can call tools

A pure text model can be wrong and annoying. A tool-calling model can be wrong and destructive. The architecture shifts from "response quality" to "decision quality + execution safety."

Treat tool use like distributed systems design: explicit contracts, bounded retries, idempotency, and observability.

Decision sequence before any tool call

Sequence A

Classify intent

Is this question-answering, retrieval, transaction, or orchestration?

Sequence B

Check confidence and prerequisites

Do we have enough context and required identifiers to safely call a tool?

Sequence C

Pick minimal tool set

Call the fewest tools required for this turn. More tools equals more failure points.

Sequence D

Execute with guardrails

Use timeouts, retries, and allowlists; log every call with correlation IDs.

Sequence E

Summarize and verify

Return result plus assumptions and next action if uncertainty remains.

Pattern matrix for function-calling systems

Pattern Use when Core guardrail Common mistake
Single-call deterministic One API lookup is enough Strict schema for tool args Letting model infer missing IDs
Plan-then-execute Multi-step workflows Human-readable plan checkpoint Executing steps without plan validation
Read-before-write State-changing operations Fetch latest state before write Blind update against stale data
Tool fallback chain Primary tool intermittently fails Bounded retry + fallback tool Infinite retries with growing latency
Two-phase confirmation Financial or destructive action User confirmation before commit Auto-committing critical actions

Prompt contract for tool-use discipline

You are an assistant with tool access.

Tool policy:
1) Never call a write tool unless user intent is explicit.
2) If required parameters are missing, ask a concise clarification question.
3) Prefer read tools first to verify current state.
4) For high-impact operations, require confirmation token: "CONFIRM_WRITE".
5) If tool call fails twice, stop and return fallback instructions.

Output behavior:
- Before tool call: state which tool and why (one sentence).
- After tool call: return result, confidence, and next suggested step.

Tool schema design examples

Example 1: Safe ticket update tool

{
  "name": "update_ticket",
  "description": "Update support ticket fields",
  "parameters": {
    "type": "object",
    "required": ["ticket_id", "changes", "confirmation_token"],
    "properties": {
      "ticket_id": { "type": "string" },
      "changes": {
        "type": "object",
        "additionalProperties": false,
        "properties": {
          "priority": { "type": "string", "enum": ["low", "medium", "high", "critical"] },
          "assignee": { "type": "string" },
          "status": { "type": "string", "enum": ["open", "in_progress", "closed"] }
        }
      },
      "confirmation_token": { "type": "string", "const": "CONFIRM_WRITE" }
    },
    "additionalProperties": false
  }
}

Execution loop pattern with retries and idempotency

async function runToolLoop(userRequest) {
  const plan = await model.plan(userRequest);
  assertAllowedTools(plan.tools);

  for (const step of plan.steps) {
    const args = validateArgs(step.tool, step.args);
    const callId = createIdempotencyKey(step);

    const result = await callToolWithRetry(step.tool, args, {
      retries: 2,
      timeoutMs: 8000,
      idempotencyKey: callId,
    });

    if (!result.ok) {
      return safeFallback(step, result.error);
    }

    addToContext(result.payload);
  }

  return summarizeFinalAnswer();
}

Six concrete scenarios

Scenario 01

Support assistant triaging account issue

Flow: `get_customer_profile` → `get_recent_tickets` → suggest action. No write tools needed.

Scenario 02

Refund request workflow

Flow: read billing history → compute eligibility → ask confirmation → write refund request.

Scenario 03

Engineering deployment assistant

Flow: fetch environment health → verify release gates → propose deploy command, never auto-execute.

Scenario 04

Security triage bot

Flow: gather alerts → correlate incidents → recommend containment checklist. Block any direct network changes without human approval.

Scenario 05

Sales operations assistant

Flow: fetch account context → draft next actions → optionally create CRM task after user confirmation.

Scenario 06

Procurement agent with policy checks

Flow: read policy document → validate request threshold → route for approval. Never bypass approval chain via direct tool call.

Observability and incident response for tool calls

  • Log every tool call with `request_id`, `tool_name`, and argument hash.
  • Track success rate and p95 latency per tool endpoint.
  • Alert on retry spikes and rising fallback rates.
  • Record idempotency key usage for duplicate prevention audits.
  • Persist full call chain for post-incident replay.

Anti-patterns that break production quickly

Tool spam loop

Model calls many tools in sequence with weak relevance. Fix by requiring explicit justification per call.

Unbounded retries

Latency explodes during outages. Fix with bounded retries and deterministic fallback.

Write-first behavior

State corruption risk rises. Fix with read-before-write and confirmation gates.

Implementation challenge

Pick one workflow that currently mixes read and write actions. Refactor it into explicit phases: plan, read, validate, confirm, write, summarize. Add idempotency keys and retry caps. If your incident rate drops and response time remains stable, you are on the right architecture path.
