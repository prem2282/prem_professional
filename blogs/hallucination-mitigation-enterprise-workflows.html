Hallucination Mitigation in Enterprise Workflows

Enterprise teams do not need zero hallucinations—they need bounded risk: answers that are grounded, cite sources, or abstain when evidence is thin.

Legal asked whether our copilot could "guarantee truth." I said no model can—but we can guarantee process: retrieval-first for facts, structured outputs for machine actions, human review for high-impact decisions, and telemetry when abstain rates spike.

Hallucination is not one bug—it is a family of failures: invented facts, wrong retrieval, over-confident synthesis, and actions taken without authority.

Four types of "hallucination" in enterprise tickets

TypeWhat users seeRoot cause (usually)
Intrinsic fabricationPlausible policy that does not existNo retrieval; model fills gaps
Contextual misreadRight doc, wrong clauseChunking / ranking
Stale truthOld process cited as currentIndex freshness, version tags
Action hallucinationTool called with invented IDsWeak schemas + no validator

Fixing all four with one knob (temperature, fine-tune, bigger model) rarely works. Map the incident type first.

Workflow patterns that scale

  • Retrieve → generate → verify: Second pass checks claim against sources before showing to user.
  • Structured actions only: Free text for humans; JSON/tool calls for systems—validators catch invented fields.
  • Human-in-the-loop tiers: Auto-approve low-risk intents; queue refunds, legal, and medical adjacency.
  • Canary prompts: Synthetic questions with known answers in prod; alert when faithfulness drops.
  • Source versioning: Attach `doc_id` + `version` + `effective_date` to every chunk; refuse when user asks about dates before coverage.

Verify pass (lightweight)

Split answer into atomic claims. For each claim, check whether any retrieved sentence entails it (NLI model or keyword overlap + embedding). Fail closed: if >1 claim fails, abstain or show draft with warning banner. 

class AnswerPolicy:
    min_retrieval_score: float = 0.72
    require_citation: bool = True

    def respond(self, query, chunks):
        if not chunks or chunks[0].score < self.min_retrieval_score:
            return Abstain("No authoritative source found.")
        draft = generate(query, chunks, cite=True)
        if not verify_claims(draft, chunks):
            return Abstain("Could not verify against sources.")
        return draft

What to measure

Track abstain rate, citation coverage, human override rate, and "confident wrong" incidents per 1k sessions. A rising abstain rate often means retrieval broke—not that the model got worse.

  • Citation precision: Of cited chunks, what % actually support the claim (human audit sample)?
  • Abstain precision: When we refuse, how often was the user question unanswerable vs retriever failure?
  • Override after accept: Support edits post-answer—leading indicator of silent wrong.

Incident taxonomy for postmortems

  1. HW-1: Wrong chunk at rank 1 → retrieval/rerank/chunking.
  2. HW-2: Right chunk, model ignored → prompt packing or instruction conflict.
  3. HW-3: No chunk available, model invented → abstain threshold too low.
  4. HW-4: Tool/action error → schema validation, ACL, idempotency.

Your implementation challenge

Label 50 production transcripts with HW-1…4. If >40% are HW-1, invest in retrieval before another model upgrade. If HW-3 dominates, raise abstain threshold and improve empty-state UX.
