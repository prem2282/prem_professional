Chunking and Embedding Strategies

Your vector database is only as smart as the pieces you feed it. Chunking decides what those pieces mean; embeddings decide how they are found. This workshop walks through both—with labs you can touch.

I have debugged RAG systems that retrieved the right document but the wrong paragraph. The embedding was doing its job; the chunk boundary had amputated the rule we needed. That is why I treat chunking and embedding as one design conversation—not two checklist items.

Why chunking is a product decision, not a utility setting

Experiment A

Chunk size sets the granularity of meaning. Small chunks improve precision for pinpoint facts but lose surrounding qualifiers. Large chunks preserve narrative but dilute embeddings with mixed topics—search gets "muddy."

Experiment B

Overlap is insurance against boundary bad luck. Fifteen to twenty percent overlap is a common starting point for prose policies; tables and code often need structure-aware chunking instead of blind sliding windows.

Chunking strategies I see in production

Fixed window

Fast, predictable. Risk: cuts through tables and numbered lists. Best for homogeneous prose.

Structure-aware

Split on headings, markdown, HTML sections, or PDF layout blocks. Best for docs with clear outline.

Semantic / recursive

Merge/split until units are "about one idea." Higher compute at index time; often worth it for messy corpora.

Parent–child chunks

Retrieve small chunk, inject parent section for generation. Balances search precision with answer context.

Embedding strategy: what matters beyond "pick a model"

  • Same embedder at index and query time—non-negotiable.
  • Normalize vectors when using cosine similarity in your store.
  • Enrich with metadata: title, section, product, version, ACL—filter before vector math.
  • Evaluate on your data: benchmark queries from real tickets, not generic sentence pairs.
  • Hybrid search when exact tokens matter (SKUs, error codes, policy IDs).

Failure gallery (from real postmortems)

Split table

Row-level rules separated from headers. Model cites "30 days" without seeing it applies only to tier A.

Boilerplate pollution

Headers/footers embedded in every chunk. "Confidential" and copyright lines become top matches.

Model swap without re-embed

New embedder, old index. Recall collapses; team blames "RAG regression."

Language mix

Multilingual corpus, English-only embedder. Non-English queries retrieve random neighbors.

Production checklist

  • Define target chunk size from median paragraph + token budget.
  • Log chunk_id with every retrieval and every answer.
  • Version embedder + index together; re-embed on change.
  • Run offline recall@K on 50–200 real queries before launch.
  • Inspect top failures manually—patterns beat aggregate metrics.

Your workshop homework

Take one production document that caused a wrong answer. Run it through Split Lab with three modes and two sizes. Note which configuration would have kept the critical sentence intact. Then write one sentence describing the metadata filter you should have applied before vector search.

Chunking shapes meaning. Embeddings find meaning. If the shape is wrong, search will look confident anyway.
