Caching Strategies for AI Responses

Caching LLM output is not like caching a product page. The same words can mean different things after a policy update, and “close enough” semantic matches can be wrong enough to hurt.

Why we cache anyway

After latency and cost posts, caching is the lever that hits both—but only if you know what you are willing to reuse. I treat caches as a stack of increasing reach and increasing staleness risk, not a single Redis key.

Four layers on the request path

flowchart LR
  Q[Query] --> E[Exact hash]
  E -->|miss| S[Semantic similarity]
  S -->|miss| P[Provider prompt cache]
  P -->|miss| L[Full LLM call]
  L --> W[Write-through cache]

Exact match — hash of normalized prompt + model + tool schema version. Safest, lowest hit rate on paraphrases.

Semantic cache — embedding similarity above threshold; great for FAQs, dangerous for regulated answers unless versioned.

Prompt / prefix cache — provider caches long static system prompts; saves input cost and TTFT, not full answer reuse.

Retrieval cache — cache chunk IDs per query hash; avoids re-search, still runs generation.

Invalidation you cannot skip

  • Corpus version — bump kb_v in cache key on every ingest.
  • Model / prompt version — system prompt hash in key.
  • Tool schema version — agents must not reuse answers across tool changes.
  • TTL by risk class — public FAQ 24h; HR policy 15m; financial 0 semantic cache.

When not to cache

Personalized streams, real-time inventory, medical dosing, or anything where the answer must reflect seconds-fresh data—use retrieval cache only, or none. Log cache hits with cache_key_version for audits.

Challenge

Classify your top 20 queries by repeat rate. For repeats > 30%, pilot exact cache; for paraphrase clusters, pilot semantic with 0.94 threshold and corpus version key. Measure hit rate, wrong-answer reports, and p95 for two weeks.
