Context Window Management

The context window is not memory—it is a shared budget every layer competes for. Run out of room and the model does not "forget politely"; it drops what you packed last, or truncates mid-thought.

Field note from a copilot rollout: support tickets spiked after we doubled RAG chunk count. Latency was fine. Answers got worse. We had evicted the last three turns of conversation to make room.

Context window size is marketed in bold on model cards—128k, 200k, 1M. In production, your effective window is smaller: what remains after system prompts, tool JSON, safety wrappers, and the output you must reserve. Treat the window like a shared cluster budget: every team that "just adds one more chunk" is spending someone else's tokens.

What actually fills the window

Every request is a stack: system instructions, tool schemas, retrieved documents, chat history, the user message, and reserved space for the model's reply. Production failures often look like "the model ignored policy" when policy was simply truncated off the end of the prompt.

Fixed

System prompt + tool definitions rarely shrink per request—budget them first.

Variable

RAG chunks and history grow with every feature flag someone enables.

Reserved

Output max tokens are not free—they subtract from what you can send.

Who gets evicted first?

When you exceed budget, something drops. Your packer should encode explicit priority—not leave it to provider-default middle truncation. Reorder the list below to reflect your policy; the simulator drops from the bottom first when over budget.

Counting tokens like finance, not like guesswork

Use the same tokenizer family your billing uses (often via the provider's counter API). Approximations like "chars ÷ 4" drift on code, CJK, and JSON. Log per-layer counts on every request—when quality shifts without a deploy, token histograms often explain it.

  • Pre-flight check: Before retrieval, compute fixed overhead; cap retrieve_k so RAG cannot consume more than its slice.
  • Post-flight audit: Compare estimated vs billed tokens weekly; surprises usually mean duplicate history or tool results re-injected every turn.
  • Attachment policy: PDFs pasted into chat can dwarf your entire RAG budget—route large files to object storage + selective extract.

History: the silent budget thief

Multi-turn sessions feel cheap until turn twelve. Naive "send full history" scales linearly; summarization scales sublinearly if you do it right.

Pattern: rolling summary

Every N turns, summarize turns 1…N−2 into a compact state block; keep turns N−1 and N verbatim for local coherence. Store summary + recent turns in session state—not re-summarize from scratch each request (that drifts). 

# After turn 8, compress turns 1–5 into summary (once)
session.summary = summarize(turns[1:5], max_tokens=800)
prompt_history = [session.summary] + turns[6:8]  # recent verbatim

Multi-agent and tool results multiply fast

Agent loops append tool outputs to context every step. A five-step agent with 2k tokens per observation adds 10k before the model answers. Cap observation size, truncate tool JSON to relevant fields, and consider externalizing large results to blob refs the model can fetch selectively.

Packing strategies that survive feature creep

  • Budget-first packing: Allocate fixed slices before retrieval; never let RAG expand into output reserve.
  • Summarize-then-pack history: Rolling summaries beat naive "last N messages" when sessions run long.
  • Relevance-ordered RAG: Drop lowest-scoring chunks first, not the chunks appended last.
  • Tool schema diet: One mega-schema per request can cost more than your top five chunks combined.
def pack_prompt(budget: TokenBudget, layers: Layers) -> Prompt:
    fixed = layers.system + layers.tools + layers.output_reserve
    remaining = budget.total - fixed
    if remaining <= 0:
        raise BudgetError("fixed layers exceed window")

    history = trim_history(layers.history, max_tokens=remaining * 0.35)
    remaining -= count_tokens(history)

    rag = trim_rag_by_score(layers.rag, max_tokens=remaining * 0.55)
    remaining -= count_tokens(rag)

    user = layers.user  # never truncate without explicit UX consent
    if count_tokens(user) > remaining:
        raise BudgetError("user message does not fit")

    return Prompt(system=layers.system, tools=layers.tools,
                  history=history, rag=rag, user=user)

Truncation traps

Silent middle truncation (some APIs drop center context) destroys multi-turn debugging. Log token counts per layer on every request. When headroom goes negative, prefer structured refusal ("session too long—start fresh") over partial prompts the model hallucinates through.

Observability checklist

  1. Dashboard: p50/p95 tokens by layer (system, tools, RAG, history, user, completion).
  2. Alert when truncation flag or "context overflow" errors exceed baseline.
  3. Sample 20 sessions/week where users re-ask the same question—often a sign context was lost.

Your implementation challenge

Export one production trace with token breakdown. Identify the top two layers by size. Set hard caps for those layers and re-run 50 golden prompts. If task success holds and cost drops, you found real slack—not theoretical optimization.
