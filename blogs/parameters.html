Parameters Playbook: A Field Memo for LLM Teams

This is not a "what is temperature" tutorial. It is the memo I give teams when outputs are unstable, costs are drifting, and everyone keeps asking whether the model got worse.

Quick diagnosis before you touch any knob

Most bad tuning starts too early. Teams change temperature because answers "feel odd," but they have not checked retrieval quality, prompt collisions, or output truncation. Parameters are multipliers, not magic. They amplify whatever input system you already built.

If you cannot describe your failure pattern in one sentence, you are not ready to tune.

Parameter cards I actually use

Temperature

Controls variation. High values help ideation; low values stabilize policy-heavy outputs and structured tasks.

Top_p

Controls token candidate breadth. Use it with intention; avoid random combinations with high temperature unless creativity is the explicit goal.

Max output tokens

Primary latency and cost lever. Too low truncates thought, too high invites rambling and spend spikes.

Penalties and stop conditions

Useful for repetition management and workflow boundaries, especially in tool-calling and multi-step responses.

Incident log: five failures and what fixed them

Incident Observed symptom Adjustment Result
Compliance assistant drift Different tone and policy emphasis per run Lower temperature + strict output schema Consistent answers across repeated prompts
Sales copilot verbosity Long, fluffy answers, high latency Reduce max output tokens Faster outputs, clearer takeaways
Creative drafts too similar Repetitive phrasing across variants Increase temperature and top_p modestly Wider idea spread with acceptable quality
Agent loop unpredictability Tool call sequencing became erratic Move to constrained profile for planning step More stable multi-step execution
Support bot cost surge Daily token cost climbed 40% Profile caps + shorter fallback responses Cost normalized without major CSAT drop

Tuning runbook (the sequence matters)

  • Start with one task category and define pass/fail examples.
  • Freeze prompt and retrieval settings before parameter experiments.
  • Change one parameter at a time and log a version tag.
  • Review quality, latency, and cost together, never in isolation.
  • Promote only configurations that improve at least two metrics without harming the third.
const POLICY = {
  qa_strict: { temperature: 0.15, top_p: 0.9, max_output_tokens: 500 },
  assistant_balanced: { temperature: 0.35, top_p: 0.95, max_output_tokens: 750 },
  ideation: { temperature: 0.8, top_p: 1.0, max_output_tokens: 1100 },
};

const selected = POLICY[routeIntent(intent)];
const response = await llm.responses.create({
  model: "gpt-4.1-mini",
  input: prompt,
  ...selected, // Use policy by workload, not by team preference.
});

FAQ I hear from engineering leaders

Should we set one company-wide default?

No. Keep one baseline profile, but branch by endpoint intent. Shared defaults are fine; shared behavior is usually not.

What gives the quickest cost win?

Audit max output token caps and retry behavior first. Those two usually produce immediate savings.

When do we revisit parameter policy?

Whenever prompt strategy, retrieval corpus, or user task mix changes significantly. Parameter policy should evolve with the product.
