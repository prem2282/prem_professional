RAG Fundamentals for Production

The model did not hallucinate. It faithfully summarized the wrong paragraph. That is the story of most "RAG failures" I investigate—and why retrieval design matters more than model upgrades.

Retrieval-Augmented Generation is not "plug in a vector DB and hope." It is a pipeline with failure points you can measure.

Case open: the confident wrong answer

A enterprise copilot told a customer they were eligible for a full refund at day 45. Policy said the opposite. Leadership asked if we should switch to a "smarter model." We pulled the trace: retrieval had returned a marketing FAQ and an old policy draft from 2022. The LLM did exactly what we trained it to do—answer from context.

The bug lived in chunk boundaries, stale indexing, and top-K set too high without reranking. No amount of temperature tuning would fix that.

What RAG is—and what it is not

RAG is: retrieve relevant evidence, pack it into a bounded context, then generate an answer grounded in that evidence.

RAG is not: a guarantee of truth. It is not a substitute for access control, evaluation, or fresh data. It does not fix a vague system prompt.

Think of RAG as giving the model a closed-book exam with allowed notes—if you slip the wrong notes under the desk, the student will ace the wrong exam.

The production pipeline, stage by stage

flowchart LR
  IN[Ingest + normalize] --> CH[Chunk]
  CH --> EM[Embed + index]
  EM --> RT[Retrieve top-K]
  RT --> RR[Rerank]
  RR --> PK[Pack context]
  PK --> GEN[LLM generate]
  GEN --> VF[Verify + cite]

1. Ingest & normalize

PDFs, HTML, tickets, wikis—cleaned to text with stable IDs and source metadata. Garbage in becomes unfindable garbage out.

2. Chunk

Split documents into retrieval units. Too large mixes topics; too small loses context. Headers and section boundaries matter.

3. Embed & index

Vectors (and often keyword/BM25 hybrid) land in a store with version tags. Stale index = confident outdated answers.

4. Retrieve

Query embedding finds nearest neighbors. High top-K without filtering floods the prompt with noise.

5. Rerank

Cross-encoder or LLM reranker re-orders candidates. Often the highest ROI step teams skip.

6. Pack & generate

Fit the best chunks into token budget; model answers with instructions to cite or abstain.

7. Verify

Citation checks, confidence thresholds, escalation to human when evidence is weak.

Evidence locker: failures I see in the wild

Exhibit A

Chunk boundary split. Refund policy table cut mid-row. Retrieval returns half a rule; model infers the rest.

Exhibit B

Semantic collision. "Termination" in HR doc ranks above "termination clause" in contract because embedding similarity is deceptively close.

Exhibit C

Stale corpus. Index not refreshed after policy v3 launch. Answers cite v2; legal team cites v3. Trust dies fast.

Exhibit D

Top-K inflation. Team raises K from 5 to 20 after one miss. Precision collapses; latency and cost rise; misses continue for different queries.

Exhibit E

No abstention path. Weak retrieval still triggers fluent generation. Users prefer "I don't know" over wrong policy.

Design rules that survive production

  • Measure retrieval precision@K and answer groundedness separately.
  • Version your index; tie answers to document version IDs in logs.
  • Prefer hybrid search when keywords matter (SKUs, policy IDs, error codes).
  • Rerank before you widen K.
  • Pack by token budget, not chunk count.
  • Ship an explicit insufficient evidence response path.

When RAG is the wrong tool

Skip RAG when you need deterministic policy execution (use rules engine + LLM for language only), when data changes second-by-second without index pipeline, or when the task is pure transformation of provided input. Adding vectors adds latency, ops burden, and new failure modes—earn that complexity.

Minimal production-shaped code

async function answerWithRag(query, userCtx) {
  const candidates = await hybridSearch({
    query,
    topK: 24,
    filters: aclFilter(userCtx), // tenant + role scoping
  });

  const ranked = await rerank(query, candidates.slice(0, 12));
  const packed = packByTokenBudget(ranked, { maxTokens: 3200, reserveForAnswer: 700 });

  if (packed.bestScore < GROUNDING_THRESHOLD) {
    return { type: "abstain", message: "Insufficient policy evidence." };
  }

  return llm.generate({
    system: "Answer only from CONTEXT. Cite chunk_id per claim.",
    context: packed.chunks,
    query,
  });
}

Your mission

In the Investigator above, turn off reranking and fresh index, set Top-K loose, and run the query. Read the simulated answer. Then turn on every guardrail and run again. That gap is what your users feel when retrieval is treated as "solved" because vectors are enabled.

Build RAG like infrastructure: observable, versioned, and testable—not like a feature flag.
