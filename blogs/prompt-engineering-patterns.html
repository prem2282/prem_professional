Prompt Engineering Patterns: A Practical Pattern Library

I keep this as a reference when teams ask why outputs are inconsistent, verbose, or off-target. Instead of generic advice, this guide gives reusable patterns with many concrete examples. Use it like a cookbook: pick a pattern, apply the template, run your evals, then iterate.

Pattern Pack

Each pattern below includes why to use it, a weak prompt, and a stronger production variant. Mix patterns, but be intentional about ordering.

  P --> R[Role + objective lock]
  P --> C[Constraints + output format]
  P --> E[Few-shot examples]

Pattern 01Role + Objective Lock
Best for: consistencyRisk reduced: vague tone

Anchor the model in a specific role and explicit task objective before giving content.

Weak: "Review this PR and give feedback."
Better: "You are a senior backend reviewer. Evaluate this PR for regressions, security risk, and missing tests. Return findings ordered by severity."
Pattern 02Output Contract
Best for: toolingRisk reduced: unparseable output

Define exact output shape, field names, and mandatory sections.

Weak: "Summarize this incident."
Better: "Return JSON with keys: incident_summary, root_cause, blast_radius, immediate_fix, long_term_fix. Keep each field under 80 words."
Pattern 03Few-Shot Behavior Steering
Best for: style transferRisk reduced: format drift

Provide 1-3 short examples showing desired style and constraints.

Example Input: "Database latency rose 20%"
Example Output: "Latency increased 20% after release v2.7. Primary suspects: index miss and cache eviction spike."

Now apply the same style to: "Support queue increased by 35%"
Pattern 04Constraint Ladder
Best for: precision tasksRisk reduced: hallucinated extras

Stack constraints from most critical to least critical.

1) Use only provided context.
2) If evidence is missing, say "insufficient evidence."
3) Max 120 words.
4) Include exactly 2 bullet points.
Pattern 05Deliberate Decomposition
Best for: complex tasksRisk reduced: shallow reasoning

Break one hard prompt into explicit sub-steps and force intermediate checks.

Template: "Step 1 classify issue type. Step 2 list evidence. Step 3 propose fix options. Step 4 choose best option and justify in 3 lines."
Pattern 06Ask-Back Clarifier
Best for: ambiguous user requestsRisk reduced: wrong assumptions

Instruct the model to ask clarifying questions when confidence is low.

Template: "If requirements are ambiguous, ask up to 3 clarifying questions before generating final output."
Pattern 07Evidence-Citation Hook
Best for: factual trustRisk reduced: unsupported claims

Require every non-trivial claim to be tied to source snippets.

"For each recommendation, include: (a) claim, (b) exact evidence line, (c) confidence high/medium/low."
Pattern 08Counterexample Testing
Best for: policy promptsRisk reduced: brittle rules

Give near-miss and edge-case examples so the model learns boundaries, not just happy paths.

Example: Include one "allowed request" and one "nearly allowed but blocked" request to clarify policy edges.
Pattern 09Failure Fallback Behavior
Best for: production botsRisk reduced: silent failure

Specify exactly what to do when data is missing, tool call fails, or confidence is low.

Template: "If tool fails, return a partial answer + a retry suggestion + a precise missing-data list."
Pattern 10Persona Split by Channel
Best for: multi-surface productsRisk reduced: tone inconsistency

Use different output persona constraints for email, chat, and executive summaries.

"If channel=exec-brief: max 5 bullets, strategic tone.
If channel=support-chat: concise and empathetic.
If channel=engineering-note: technical with assumptions list."

Examples by Domain

These are complete prompts you can copy and adapt.

Example A: PR Review Assistant

You are a principal engineer reviewing a pull request.
Objectives:
1) Find behavioral regressions.
2) Identify security and data-leak risks.
3) Flag missing tests.

Output format:
- Severity: Critical/High/Medium/Low
- File: path
- Issue
- Why it matters
- Suggested fix

Constraints:
- Do not praise style choices unless they affect behavior.
- If no major issues, explicitly say "No material risks found."
- Ask one clarification if a diff is incomplete.

Example B: Customer Support Copilot

You are a support copilot for a SaaS billing product.
Use only the policy context.
If policy is missing, answer: "I need policy confirmation."

Write response in 3 blocks:
1) Direct answer (1-2 sentences)
2) Why (cite policy clause)
3) Next action for agent

Tone: calm, non-legal, non-defensive.
Max output: 120 words.

Example C: Incident Triage Assistant

Task: Analyze incident timeline and produce triage memo.

Steps:
1) Classify incident type.
2) List top 3 likely causes with evidence.
3) Propose immediate mitigation.
4) Propose follow-up checks for next 24h.

Return markdown with headings:
## Classification
## Top Causes
## Immediate Mitigation
## 24h Watchlist

If confidence below medium, ask 2 clarifying questions first.

Example D: Prompt for structured marketing drafts

Role: B2B technical content strategist.
Create 3 launch announcement variants for feature: "adaptive retrieval."

Variant constraints:
- V1: executive tone, 90-110 words
- V2: practitioner tone, 130-160 words
- V3: social post, under 70 words

Must include:
- customer problem
- what changed
- measurable benefit
- one CTA

Avoid hype words: revolutionary, game-changing, seamless.

Example E: Data extraction pattern

Extract entities from the provided contract text.
Return strict JSON array:
[
  {
    "party_name": "",
    "effective_date": "",
    "renewal_term": "",
    "termination_notice_days": 0,
    "source_quote": ""
  }
]

Rules:
- Use null when missing.
- Do not infer unstated values.
- Include source_quote for every field.

Example F: Teaching assistant pattern

You are a senior mentor teaching cloud architecture.
Given learner question and level, produce:
1) Short answer
2) Analogy
3) One pitfall
4) One practice exercise
5) One follow-up question to test understanding

Audience: junior engineer
Keep language plain and practical.

Anti-Patterns That Keep Appearing

Anti-pattern: giant all-in-one prompt with conflicting instructions.

Fix: split into system policy + task instruction + output contract.

Anti-pattern: asking for "detailed" output without length or structure constraints.

Fix: add target word range and section schema.

Anti-pattern: relying on one perfect prompt version forever.

Fix: version prompts and evaluate against a stable test set.

Three Quick Prompt Builds

Build 1: Fast deterministic answer

Role + Objective
Allowed evidence only
Output contract
Length cap
Fallback rule ("insufficient evidence")

Build 2: Balanced reasoning answer

Role + task decomposition
Evidence requirements
Decision rubric
Final concise recommendation

Build 3: Creative variant generator

Audience persona
Few-shot style anchors
Variant constraints (3-5 versions)
Banned phrases
Final self-check: distinctness across variants

Review Checklist Before Shipping

  • Is the role explicit and task objective unambiguous?
  • Is the output contract machine-parseable where needed?
  • Are constraints prioritized from critical to optional?
  • Does fallback behavior exist for low confidence/tool failure?
  • Did you test at least 10 real queries including edge cases?
  • Do you have prompt version tags and evaluation notes?

If you are building a serious product, treat prompt design as an engineering artifact: versioned, tested, and reviewed. Patterns make that possible.
