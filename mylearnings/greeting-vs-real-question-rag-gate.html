When “Hi” Cost 20,000 Tokens: Routing Greetings Before RAG

In our early GenAI customer-support pilot, the chatbot was forbidden from general knowledge—so every message, even a two-letter greeting, triggered full retrieval. The fix was not a smarter model. It was orchestration with a scope box.

The invoice hiding in a handshake

We were building a customer-support chatbot in the early GenAI era—before models were good at small talk, and long before mini and nano tiers existed as cheap routers. The product rule was strict: answers had to come from private proprietary documents only. No Wikipedia, no model parametric knowledge, no “helpful” improvisation.

That policy pushed us into a classic enterprise pattern: RAG on every user turn. Embed the question, pull top-k chunks from the vector store, pack them into the prompt, call the flagship LLM. On average, a real question consumed 20,000–30,000 input tokens—system prompt, retrieved chunks, formatting overhead. For a genuine policy or product question, that was the cost of staying grounded.

Then someone opened the widget and typed “Hi”.

The pipeline did exactly what we told it to do. Retrieval returned 30 chunks loosely related to “Hi”—headers, contact boilerplate, fragments that happened to embed near the greeting. Context assembly ballooned. The LLM burned roughly 20K tokens to produce… “Hello, how can I help you today?”

What hurt: not the quality of the reply—the unit economics of treating social openers like substantive queries in a closed-corpus bot.

RAG on every turn is like running a full database scan to shake hands. The analogy breaks if your corpus is tiny—but ours was not, and top-30 retrieval was non-negotiable for real questions.

Baseline architecture: closed corpus, open throttle

Before any fix, the happy path looked like this. Every message—turn 1 through turn N—hit the same heavy path because we could not trust the model to answer without evidence.

flowchart LR
  U[User message] --> E[Embed query]
  E --> V[Vector search top-30]
  V --> C[Context assembly]
  C --> L[Flagship LLM]
  L --> R[Reply]

Ingress: chat orchestrator receives text from the support widget.
Retrieval: embedding model + vector DB; k=30.
Context assembly: system instructions + chunk text + user message → ~20K–30K tokens for real questions.
Model: single flagship tier—no small model on the menu yet.
Invariant: no answer without retrieved evidence.

In short: we had optimized for grounding, not for intent shape. Greetings are not questions—but our pipeline could not tell the difference.

Why “Hi” still retrieved 30 chunks

Embeddings measure semantic similarity, not conversational intent. “Hi” is short, ambiguous, and sits near many vectors in a support corpus—welcome pages, chat scripts, “Say hi to our team,” FAQ intros. With top-30 retrieval, you almost always get something, even when the user asked for nothing.

The LLM then dutifully read a wall of context and generated a polite opener. Technically correct. Financially absurd.

We considered quick fixes:

  • Regex for “hi/hello” — brittle across languages, typos, and “Hi, I need help with billing.”
  • Skip RAG for all short messages — dangerous; “refund?” is short but not a greeting.
  • Cheaper model for openers — did not exist in our model catalog at the time.
  • Always-on RAG — status quo; unacceptable burn rate at pilot scale.

What looked like a token problem was really an orchestration problem: we needed a gate before retrieval, not a better answer after it.

The gate: classify intent, not answer content

We built a small in-house classifier—not to replace the LLM, but to label the first user message: greeting/social opener vs real question. Binary route:

  • Greeting → lightweight LLM reply, no RAG, fixed short template policy.
  • Real question → full retrieval pipeline unchanged.

The critical scope box: run the classifier on the first message of a conversation only.

Most greetings happen at open. Turn 3 is rarely “Hi.” Running classification on every turn would add latency and compute for marginal gain—classic over-engineering. The boundary saved us twice: less inference, simpler ops.

# Orchestrator pseudocode (first turn only)
if conversation.turn_index == 1:
    label = greeting_classifier(user_text)  # sync call
    if label == "greeting":
        return llm_reply_without_rag(user_text)
return rag_pipeline(user_text)

In short: we stopped paying for retrieval when the user had not asked anything yet.

flowchart TB
  M[First user message] --> F{Turn 1?}
  F -->|no| RAG[Full RAG path]
  F -->|yes| C{Greeting classifier}
  C -->|yes| LIGHT[LLM without retrieval]
  C -->|no| RAG
  RAG --> OUT[Reply]
  LIGHT --> OUT

Infra: why Lambda beat a always-on server

The classifier needed compute—but our pilot saw only a few hundred conversations per day. Provisioning a dedicated GPU/CPU server to stay up 24/7 for that duty cycle was ops overhead we did not want: patching, monitoring, paying for idle hours.

We packaged the model as an AWS Lambda function, invoked synchronously from the chat orchestrator on first turn only.

sequenceDiagram
  participant O as Chat orchestrator
  participant L as Lambda classifier
  participant R as RAG pipeline
  O->>L: first message text
  L-->>O: greeting | question
  alt greeting
    O->>O: LLM lite path
  else question
    O->>R: embed + retrieve + LLM
  end

Compute model: per-invocation billing vs idle baseline.
Trigger scope: once per conversation start—not per message.
Cold start: acceptable in pilot; first message already waited on network + LLM.
Ops: deploy artifact, CloudWatch logs, no cluster to babysit.
Scale path: at 10× traffic I would revisit containerized always-warm instances or swap the classifier for a managed mini-model endpoint—but at pilot volume, serverless was the apt fit.

Eye-opener: The ML model was not the innovation—the scope box (first turn only) plus duty-cycle-matched infra (Lambda for hundreds/day) was. Many teams buy GPUs; we bought a routing decision.

Impact we measured

Typical conversations ran 4–5 Q&A turns. Skipping full RAG on the opening greeting removed one heavy token event per session—roughly 20% token savings per conversation in the pilot, with infra cost for classification negligible at that volume.

That percentage is not magic—it is arithmetic. One of five turns no longer paid the 20K–30K retrieval tax. As sessions grew longer, the percentage shrinks; as users greeted repeatedly in new sessions, it grows. We tracked tokens per conversation, not per request, because finance and capacity plan at session level.

What changed when mini and nano models arrived

The ecosystem caught up. Cheap, capable small models can route intent with a single LLM call—often replacing a bespoke classifier entirely. We retired the in-house model and simplified the stack.

The lesson survived the swap: separate routing from answering, and match infra to duty cycle. The implementation medium changed; the architecture pattern did not.

If I were shipping the same product today, I would likely start with a mini-model router on turn one, keep the no-RAG path for greetings, and only rebuild custom ML if evals showed systematic misroutes on our domain vocabulary.

Takeaways

  1. Grounding policy creates traffic shape. Closed-corpus RAG makes every turn expensive by design—budget for orchestration, not just retrieval quality.
  2. Retrieval follows intent, not politeness. Embed “Hi” and you will get chunks; that is working as designed, not a bug.
  3. Scope boxes beat brute force. First-turn-only classification captured most savings without taxing every message.
  4. Infra follows volume. Lambda for hundreds/day; revisit when cold starts or concurrency become your p99 enemy.
  5. Patterns outlive implementations. Custom classifier → mini model; greeting gate stays.
Repeat in a design review: “Are we retrieving before we know the user asked a question?”
