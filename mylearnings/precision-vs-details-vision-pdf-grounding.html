Beautiful Markdown, Wrong Numbers: When Vision-Only PDF Ingest Failed Precision

A client RAG pipeline needed dense product-catalog tables from PDFs—vision models gave us gorgeous structure and catastrophic decimals. The fix was not a better prompt alone; it was giving the model a second, boring source of truth.

The demo that almost shipped a lie

We were ingesting a large set of product-catalog PDFs for a client chatbot. Pages were brutal: numerical tables, spec grids, small charts, and marketing graphics crammed into one or two pages—far more information density than a typical brochure. Users would ask precise questions: wattage, dimensions, compatibility codes, price breaks. In RAG, one wrong number is not a formatting issue; it is a business-loss event.

This was the window when vision-capable models had just become practical for document understanding. The pitch was seductive: render each PDF page to an image, send it to the vision model with a careful system prompt, receive clean markdown with tables and bullets. On a human read, the output looked publication-ready.

Then we spot-checked values against the PDF. A rating listed as 70.12 in the source became 1012 in the extracted markdown. The chatbot would later cite that chunk with confidence. Retrieval was working. Embeddings were working. The corpus was poisoned at ingest.

What looked like a retrieval-quality problem was really an extraction-precision problem wearing a presentation-quality mask.

Why this corpus was harder than “just OCR”

Classic text extractors struggle when layout is complex—merged cells, footnotes, multi-column tables, tiny fonts. Vision models shine at layout and structure: they “see” the table, infer headers, and emit readable markdown. Under the hood, though, many vision pipelines still lean on OCR-like character recognition for the actual digits. Dense pages amplify OCR failure modes:

  • Decimal collapse — the dot between 70 and 12 disappears; digits merge into 1012.
  • Column bleed — adjacent cells smear in tight grids.
  • Graphic vs data confusion — labels on charts read as table values.

Chunking and reranking cannot fix a spec sheet that was wrong before it was embedded. Garbage in, confident garbage out.

Path A — vision only (what we tried first)

flowchart LR
  PDF[PDF page] --> IMG[Page image]
  IMG --> V[Vision LLM]
  V --> MD[Markdown chunks]
  MD --> E[Embed + vector store]
  E --> RAG[Chatbot RAG]

Components: PDF rasterizer → image per page → vision model + system prompt (“extract tables, preserve units, output markdown”) → chunker → vector DB → existing RAG orchestrator.

What worked: structure, headings, bullet hierarchy, table shape, interpreting graphics in context.

What failed: numeric fidelity on packed tables—the exact fields users query most.

In short: we optimized for readability of the ingest artifact, not trustworthiness of the facts.

Path B — dual source: image + raw text

We did not abandon vision—it was too good at structure. We added a parallel, unglamorous lane: raw text extraction with PyMuPDF (any faithful text library would do). Same page, two inputs to the same vision-capable model:

  1. Page image — layout, tables, graphics, reading order.
  2. Raw extracted text — character-level ground truth for numbers and codes, however messy.
  3. System prompt — explicit instruction: use the image for structure; reconcile every numeric field against the raw text; flag conflicts.

The LLM’s job shifted from “transcribe what you see” to editor with two manuscripts—the photograph and the accountant’s ledger. When OCR inside the vision stack mangled a decimal, the raw text still said 70.12, and the model consistently preferred it for the final table.

On the complicated catalog pages we tested in QA, we reached accurate numeric extraction—the bar the client cared about. Not “pretty markdown.” Correct specs.

Eye-opener: The breakthrough was not “smarter vision.” It was refusing to let vision be the only witness for digits.

Prompt contract (sketch)

The system prompt had to be boring and enforceable—not creative.

You receive (1) a page image and (2) raw text extracted from the PDF.
Use the image for layout, tables, headings, and chart interpretation.
For every number, unit, SKU, and dimension, prefer the raw text when
image-derived values disagree. Output markdown tables. List unresolved
conflicts in a short "qa_flags" block.

We still human-sampled pages after ingest, but the dual-source contract cut systematic OCR-style errors on dense tables.

Failure modes if you copy this blindly

  • Scanned PDFs with no text layer — PyMuPDF returns empty or garbage; you need real OCR or a human pass, not a fake second source.
  • Text layer itself wrong — some PDFs embed incorrect copy; reconciliation needs checksums or spot QA.
  • Token cost — image + long raw text per page is expensive; batch ingest and cache markdown artifacts.
  • Over-trusting the reconciler — the LLM can still hallucinate a merge; keep eval hooks on critical fields.

What I’d do today

Models and vendor parsers have improved, but the pattern holds: separate presentation inference from fact grounding. For table-heavy RAG, I would still run a cheap text-layer extract alongside any vision or layout model, plus a regression suite of “known spec” cells per template family.

Downstream, I would tag chunks with page_id and extract_version so when a user reports a wrong wattage, we can re-run ingest for that page—not retrain retrieval.

Takeaways

  1. Precision and presentation diverge on dense PDFs. Judge ingest on spot-checked fields, not markdown beauty.
  2. Vision is layout intelligence, not automatic truth. OCR limitations do not disappear because the API says “vision.”
  3. Dual-source beats dual-model. One capable model with image + text often outperforms vision-only with a longer prompt.
  4. RAG cannot fix poisoned chunks. Fix extraction before embedding; retrieval quality is downstream.
  5. One wrong spec is a business event. In catalog bots, numeric QA is part of the architecture—not a nice-to-have.
Repeat in a design review: “What is the ground-truth lane for digits—and is it in the prompt context?”
